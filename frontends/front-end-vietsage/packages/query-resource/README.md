# @dangminhdev/query-resource

A transport-agnostic, capability-driven resource factory for TanStack Query v5.

The package does not call `fetch`, Axios, or a project-specific HTTP client. A repository returns normalized domain data; a resource owns query identity and cache behavior; feature hooks own UI orchestration.

## Boundaries

```text
Transport client -> Repository -> Resource -> Feature hook -> Component
```

- Repositories own endpoints, DTO mapping, response validation, and pagination normalization.
- Resources own keys, query/infinite options, mutations, prefetch, ensure, invalidation, and cache patches.
- Feature hooks own permissions, feature flags, debounced filters, navigation, and user feedback.
- Components consume hook results and commands.

The package is tied to TanStack Query v5, but it is not tied to VietSage or to an HTTP transport.

The repository is intentionally not generated by this package. Different projects can keep their own transport and DTO conventions:

```ts
const fetchRepository = {
  list: async (input: ListInput, signal?: AbortSignal) => {
    const response = await fetch(`/products?q=${input.q}`, { signal });
    const dto = (await response.json()) as ProductListDto;
    return mapProductPage(dto);
  },
};

const axiosRepository = {
  list: async (input: ListInput, signal?: AbortSignal) => {
    const response = await axios.get<ProductListDto>("/products", {
      params: input,
      signal,
    });
    return mapProductPage(response.data);
  },
};
```

Both repositories can be passed to the same resource shape because the resource only receives normalized promises.

## Capability model

`createResource` does not assume CRUD. Only declared operations exist in the returned TypeScript API.

```ts
const productResource = createResource<{ tenantId: string }>()({
  namespace: ["my-app"],
  name: "products",
  scopeKey: (scope) => ["tenant", scope.tenantId],
  queries: {
    list: defineQuery({
      inputKey: (input: ProductListInput) => [input],
      queryFn: listProducts,
    }),
    detail: defineQuery({
      inputKey: (input: { id: string }) => [input.id],
      queryFn: getProduct,
    }),
  },
  mutations: {
    create: defineMutation({
      mutationFn: createProduct,
      invalidates: [{ type: "query", operation: "list" }],
    }),
    update: defineMutation({ mutationFn: updateProduct }),
    remove: defineMutation({
      mutationFn: removeProduct,
      invalidates: [{ type: "query", operation: "list" }],
    }),
  },
});
```

A read-only module declares only its query:

```ts
const countryResource = createResource<void>()({
  namespace: ["my-app"],
  name: "countries",
  scopeKey: () => ["public"],
  queries: {
    all: defineQuery({
      inputKey: () => [],
      queryFn: countryRepository.list,
    }),
  },
});
```

Create-only or other partial modules work the same way:

```ts
const importResource = createResource<{ tenantId: string }>()({
  namespace: ["my-app"],
  name: "imports",
  scopeKey: (scope) => ["tenant", scope.tenantId],
  mutations: {
    upload: defineMutation({
      mutationFn: ({ scope, variables }: {
        scope: { tenantId: string };
        variables: FormData;
      }) => importRepository.upload(scope.tenantId, variables),
    }),
  },
});
```

There is no `extra` namespace. Domain commands such as `approve`, `cancel`, `checkIn`, or `markRead` are ordinary named mutations.

## Generated capabilities

After binding a scope, each query exposes:

- `key`, `options`
- `prefetch`, `fetch`, `ensure`
- `get`, `set`, `patch`, `patchAll`
- `invalidate`, `invalidateAll`
- `cancel`, `cancelAll`
- `remove`, `removeAll`

Infinite queries expose the same cache surface over `InfiniteData`.

Mutations expose typed TanStack mutation options. Local invalidation targets run after a successful mutation. An optional optimistic lifecycle can return one or more rollback functions:

```ts
const options = scoped.mutations.update.options({
  optimistic: ({ client, cache, variables }) =>
    cache.queries.detail.patch(client, { id: variables.id }, (current) =>
      current ? { ...current, ...variables.patch } : current,
    ),
});
```

Rollback checks whether a newer cache write occurred. If it did, the old snapshot is not written over newer data; the affected query is invalidated instead.

Declarative invalidation targets are intentionally local to the current resource and are type-checked against its declared operations. Coordinate cross-resource effects explicitly in a feature hook:

```ts
const mutation = useMutation(
  booking.mutations.checkIn.options({
    onSuccess: ({ client }) =>
      rooms.queries.list.invalidateAll(client),
  }),
);
```

## Key layout

```text
[namespace, scope, resource, capability, operation, input]
```

All response-affecting values must be represented by `scopeKey` or `inputKey`. Keys must remain JSON-serializable.

Treat scope and input values as immutable after creating query options. The generated key is a snapshot, while the repository call closes over the supplied values.

Pagination is not forced into one envelope. Offset/page inputs belong to normal queries; cursor pagination belongs to `defineInfiniteQuery`; repositories normalize each backend response into the page type selected by the feature.

## Build and verification

```bash
pnpm --filter @dangminhdev/query-resource run build
pnpm --filter @dangminhdev/query-resource run typecheck
pnpm --filter @dangminhdev/query-resource run typecheck:consumer
pnpm --filter @dangminhdev/query-resource run test
pnpm --filter @dangminhdev/query-resource pack
```

`typecheck:consumer` verifies the emitted declarations through NodeNext package resolution, not only from source. `prepack` rebuilds `dist` before a tarball is created.
